装饰者模式-咖啡售卖

装饰者模式：动态地将责任附加到对象上，若要扩展功能，装饰者提供了比继承更有弹性的替代方案。
遵循的设计原则是：类应该对扩展开放，对修改关闭

  1. 装饰者和被装饰对象有相同的类型
  2. 可以用一个或多个装饰者包装一个对象
  3. 在任何需要原始对象(被包装的)的场合，可以用装饰过的对象代替

  4. 对象可以在任何时候被装饰，可以在运行时动态地、不限量地用装饰者来装饰对象

NIO

关于 Java NIO 的详解 CSDN 上有详细解释
NIO即新的输入输出，这个库是在JDK1.4中才引入的。它在标准java代码中提供了高速的面向块的IO操作

概念

  1. IO 输入输出
    使用流的方式来处理IO，所有的IO都被视作是单个字节的移动，通过stream对象一次移动一个字节，
    流IO负责把对象与字节的相互转换
  2. NIO (NEW IO)
    这个库是在JDK1.4中引入的。NIO和IO有相同的作用和目的，但实现方式不同，NIO主要用到的是块，
    所以NIO的效率要比IO高很多

    流与块的比较

    NIO和IO最大的区别是数据打包和传输方式。IO是以流的方式处理数据，而NIO是以块的方式处理数据
    面向流的IO 一次一个字节的处理数据，一个输入流产生一个字节，一个输出流就消费一个字节,通常处理的很慢
    面向块的IO 系统以块的形式处理数据。每一个操作都在一步中产生或消费一个数据块。按块要比按流快的多

观察者模式

观察者模式 = 订阅者 + 发布者

  1. 报社的业务就是出版报纸
  2. 向某家报社订阅报纸，只要他们有新报纸出版，就会给你送，只要你是他们的订户，你就会一直收到报纸。
  3. 当你不想再看报纸的时候，取消订阅，他们就不会再送新报纸来。
  4. 只要报社还在运营，就会一直有人向他们订阅或者取消订阅报纸。

观察者模式定义了对象之间的一对多依赖，当一个对象改变状态时，它的所有依赖者都会收到通知并自动更新。
观察者模式提供了一种对象设计，让主题和观察者之间松耦合
主题(Observable)用一个共同的接口来更新观察者
可从被观察者处推(push)或拉(pull)数据
有多个观察者时，不可以依赖特定的通知次序
注意 java.utl.Observable 实现上所带来的一些问题:

  1. Observable 是一个类，必须设计一个类继承它，限制了可复用性
  2. Observable 将关键方法保护起来; 如setChanged() 被保护起来，除非继承自 Observable，否则无法
    创建 Observable 实例并组合到自己的对象中来。这一设计违反了”多用组合，少用继承”原则
